## Design refinement (post-create discussion)

Two tweaks landed in conversation that this ticket should honour:

**OAuth is opt-in, not always-on.** Local stdio mode does NOT require auth (matches today's dogfood behaviour — single user on localhost). The trigger is config:
- `auth.providers` block empty (or whole `auth` block absent) → no auth, server runs in "single-user trusted localhost" mode as today.
- `auth.providers` block populated → OAuth required for the web UI; `requireMember` middleware activates per [[auth-middleware-route-guards-csrf-reconciliation]].
- Document this clearly in the settings page + README. Remote `serve` deployments should be told to populate the block.

**GitHub OAuth is the primary provider.** Reason: the OAuth flow yields a GitHub access token that the server reuses for the Phase 3 git-linkage feature (commit/PR/release indexing via GitHub API). One OAuth, two purposes. Google OAuth stays as a secondary option for users who don't host on GitHub.
- Store the GitHub access token (encrypted with the server secret) on the `User` record. Refresh-token flow as standard.
- The git-linkage indexer reads this token to call GitHub API on behalf of the user.

These are clarifications, not rescopes — the existing acceptance criteria stand. Add a "Single-user mode (no auth providers configured)" code path that short-circuits the middleware to a synthetic local user, and a config validation step that warns loudly if `auth.base_url` is set but no providers are configured.
