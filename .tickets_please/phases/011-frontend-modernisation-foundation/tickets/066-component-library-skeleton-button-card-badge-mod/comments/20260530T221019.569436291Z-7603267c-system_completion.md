Learnings:
- **`@` in templ prose explodes the parser silently.** `<code>@Card</code>` inside a children block was parsed as a component invocation, generating `Card.Render(ctx, ...)` (bare-type method call) and failing `go build` with "Card.Render undefined". Templ doesn't catch this at generate time — `templ generate` succeeds and only the Go compiler complains. Fix: HTML-entity-escape with `&#64;Card` (or any non-`@` formulation) in prose. The `@Badge(...)` invocations inside `<td>…</td>` ARE valid components — the failure mode is specifically a bare `@Identifier` followed by something that isn't `(…)` or `{…}`.

- **Templ component signature is `func Name(props) templ.Component`** — so `@Name(props)` resolves to a Component and renders normally inside another templ. Critically: composing components inside `<td>` cells works the same as inside any other children block. The playground's `@Table(...) { <tr><td>@Badge(...)</td></tr> }` is the canonical "composed cell" example.

- **`templ.Attributes` passthrough is essential.** Without it, anchoring `data-dialog="…"` onto a Button means wrapping the button in an outer element or smuggling a child `<span data-dialog="…">` and walking back up via `.closest()` — both fragile. The `Attrs templ.Attributes` field with `{ props.Attrs... }` spread in the .templ is the clean answer; should be on every component that might need hx-*/data-*/aria-* hooks. Added to Button now; other components can grow it on demand. Pattern: declare Attrs as the LAST field of the Props struct so call sites that don't need it can `{Variant: …, Label: …}` and omit Attrs entirely.

- **Mixing .go and .templ in the same package is the standard templ workflow.** Helper functions (`buttonClass`, `badgeClass`), variant constants, and prop structs all live in sibling .go files; the .templ files just consume them. The templ generator emits `*_templ.go` in the same package, so package-level identifiers are visible across both file types — no import dance.

- **Tailwind v4's @source scan is recursive into subdirs.** `@source "../../components"` in `_src/app.css` (set up by [[tailwind-v4]]) picks up the new `components/ui/*.templ` files automatically — no @source addition was needed for the new package. The class names `max-w-[64rem]`, `mx-auto`, `my-8`, `px-6` referenced in `playground.templ` materialise in the next `make css` without further config.

- **Variant-class-table in Go, NOT in .templ.** Putting the switch in a Go helper (`buttonClass(p ButtonProps) string`) means renaming a variant or adding a size is a one-file change — call sites pass typed constants. Inlining the switch in the .templ would scatter class strings across template branches. Same pattern as shadcn/ui's `cva`; the Go version is just a `switch` + `strings.Join`.

- **The playground IS the test.** No Go test files in `components/ui/` because every component is exercised end-to-end on `/_dev/components` — and the smoke I ran (curl + grep-by-class) is essentially a snapshot test on the rendered HTML. A future ticket could promote the smoke into a `_test.go` that hits httptest.NewServer + the playground handler and asserts class presence; the pattern shown here ports cleanly.

- **Native `<dialog>` open/close JS is the canonical pattern in this codebase** (already established by [[dialog-modals]]). The playground inlines the same 3 selectors — `[data-dialog]` / `[data-dialog-close]` / `dialog.modal click-outside` — so the modal exercises end-to-end in isolation. When pages migrate, the JS should be hoisted into a single `_layout` script (probably the Datastar runtime ticket) so each page doesn't re-emit it.

- **Dev-route gating remains a one-line addition.** `if deps.Dev { mux.Handle("GET /_dev/components", wrap(a.handleComponentsPlayground)) }` next to the existing templ-hello line. Production builds (no `-dev` flag) return 404 on both, confirmed by the smoke run.
