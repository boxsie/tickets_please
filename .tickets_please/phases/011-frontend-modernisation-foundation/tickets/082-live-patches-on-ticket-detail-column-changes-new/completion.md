## Learnings
Datastar SSE subscribe is `data-on-load="@get('/sse?topics=...')"` in this bundle (Datastar v1), NOT `@sse(...)` as the ticket body suggested — `@sse` was an older alpha action. The playground (ui/playground.templ) is the reference. @get on an SSE response keeps the stream open and applies datastar-patch-elements/-signals frames as they arrive. Patch-by-selector design: the SSE handler renders patches keyed by topic-class, and a patch whose selector isn't on the receiving page is a harmless no-op — so the same ticket: delivery can carry ticket-detail patches that other pages ignore. Stable-id discipline matters: the inline status badge had only a class (badge-{column}) which CHANGES with the column, so it can't be a morph target — extracted it into StatusBadge wrapped in a fixed id="ticket-status". Same for the action cluster (#ticket-actions) so completing a ticket swaps in the frozen button set. The archived badge wrapper is always emitted (empty when not archived) so the #ticket-archived selector exists to morph against. renderTicketPatch re-reads authoritative state via GetTicket/ListComments rather than trusting the event payload — a since-deleted ticket just yields no patch, and the streamed comment row is byte-identical to the server-rendered one because it reuses the same buildCommentRow/toTemplRow path. The planned internal/web/sse/patch.go from the ticket became sse.PatchElements/PatchSignals builders (in sse.go, #080) plus the web-package renderer (needs *app for svc lookups, can't live in sse/). Toast auto-dismiss is pure CSS (toast-fade keyframes) — no JS; the faded element lingers invisibly in #ticket-toasts, acceptable for v1. To test streaming SSE against a real svc, sseTestServerDeps must call SetPublisher(bus) — freshDeps alone leaves the Nop publisher so mutations wouldn't fan out.
