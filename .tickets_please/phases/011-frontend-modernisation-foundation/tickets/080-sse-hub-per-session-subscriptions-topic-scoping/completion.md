## Learnings
Key design call: split the realtime stack in two to avoid an svc→web import cycle and to keep the typed events HTML-free. internal/eventbus holds the typed Bus (svc imports it as Publisher); internal/web/sse holds only the SSE framing + Datastar patch builders (web imports both). The patch *renderer* lives in the web package (sse_render.go) not in sse/, because rendering needs templ components — putting it in sse/ would be fine for cycle reasons (components don't import sse) but renderDelivery needs the *app for svc lookups, so web is the natural home; #082's planned internal/web/sse/patch.go becomes pure frame builders instead. Atomic replay is the subtle part: Subscribe snapshots ring events with seq>lastSeq AND registers the live subscription under the same mutex that Publish uses to assign seq — so a live event published after Subscribe always has a higher seq than every replayed event, giving exactly-once ordered delivery with no gap. Don't try to push replay through the live channel (interleaving races); return it as a plain slice the handler drains first. Backpressure: never close the live channel from Publish/cancel (send-on-closed panic) — signal via a separate done channel closed once (sync.Once); the handler selects on done and returns, client reconnects with Last-Event-ID. phase: topic auth is pragmatic — a phase id can't cheaply resolve to a project without the project, so a phase topic is only honored when the connection also carries an authorized project topic (phase-detail always pairs them); documented rather than doing a project-wide phase scan. Datastar v1 SSE protocol: two event names (datastar-patch-elements with mode/selector/elements lines, datastar-patch-signals with a `signals {json}` line) — the dev-ping in handlers_dev.go is the reference for the elements frame shape. The dev SSE-ping smoke button still works by publishing a synthetic KindAgentSeen on global:agents tagged AgentID="dev-ping" which renderAgentPatch routes to #sse-target.
