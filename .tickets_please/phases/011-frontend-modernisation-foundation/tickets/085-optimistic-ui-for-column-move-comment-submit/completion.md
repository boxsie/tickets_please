## Work summary
Optimistic UI on the ticket-detail move modal + comment form, plus a fix for the #082 double-render.

Plumbing: eventbus.Event gained a ClientID field; svc.WithClientID/ClientIDFrom carry an idempotency key on the context; CreateComment + MoveTicket stamp it onto their published events. The web handlers read the Idempotency-Key request header, wrap the ctx, and return 204 on success (errors → plain http.Error) — leaving the HX-Request and plain-form (303) paths untouched.

Reconciliation: sse_patch_ticket.commentAppendPatch now prepends a `mode remove #comment-pending-{ClientID}` frame when the event carries a client id, so the originating tab's optimistic placeholder is removed before the canonical row appends (no-op remove in other tabs). sse.PatchElements omits `elements` lines for an empty fragment so the remove frame is clean.

Frontend: comment_form.templ dropped its htmx beforeend wiring (it raced the SSE append → duplicate row) and gained data-optimistic-comment + data-comments-list; move_form.templ gained data-optimistic-move + data-dialog-id. New static/optimistic.js (loaded in layout.templ) progressively enhances both forms: optimistic placeholder/toast + Idempotency-Key POST + revert-on-error, with the plain form POST as a no-JS fallback. CSS: .comment-pending/.comment-error/.comment-pending-flag/.comment-retry in _src/app.css, recompiled via make css.

## Learnings
The big realisation: #082's SSE comment-append SILENTLY DOUBLE-RENDERED. The comment form was still htmx (hx-swap=beforeend) appending the POST response row, while the same tab's SSE subscription ALSO appended the row — two identical rows. Any time you add an SSE patch that appends/mutates on an action a page already handles via htmx/form, you must remove the old client-side mutation or they stack. The clean resolution is server-driven reconciliation: carry a client id (Idempotency-Key header → Event.ClientID), have the SSE echo `mode remove #pending-{cid}` then append — one authoritative path, and the optimistic placeholder gets swapped rather than duplicated. append-mode patches must stay single-sourced.

Idempotency key as a context value (svc.WithClientID) beats a new param on CreateComment/MoveTicket: zero churn at the MCP call sites, and MCP mutations just publish events with empty ClientID (renderer skips the remove frame). Same pattern as WithSessionID.

204 No Content is the right success response for an optimistic POST: the page already has (or will get) the truth via SSE, so returning HTML would just be a second source to reconcile. Keyed the new behaviour off the Idempotency-Key header so the existing 303-redirect (no-JS) and HX-Request (row partial) branches stay intact and all their tests pass untouched — additive, not a rewrite.

Progressive enhancement is the low-risk way to ship interactive JS you can't unit-test: keep the <form method=post action=...> working, intercept submit only when fetch + hooks exist, fall back to full POST otherwise. Go tests then cover the server contract (204/error/reconcile frames) and the rendered wiring attributes; the DOM behaviour rides a manual two-tab plan.

sse.PatchElements previously always emitted at least one `elements` line; for `mode remove` that's a stray empty `data: elements ` line. Guarding on html != "" (and TrimSuffix the trailing newline) keeps remove frames to just mode+selector.

Known limitation left: the move modal's <select> options are computed from the column at server-render time and are NOT a morph target (#082 deliberately only morphs the badge/#ticket-actions, not the modals). So after an optimistic move without a reload, reopening the move modal shows stale options until the next full page load. Acceptable for now; if it bites, wrap MoveForm in a stable id and add it to ticketStatePatches. Also skipped the client-side optimistic badge recolor the ticket described — the SSE morph is effectively instant on localhost, so the round-trip badge update suffices and avoids fighting the morph.
