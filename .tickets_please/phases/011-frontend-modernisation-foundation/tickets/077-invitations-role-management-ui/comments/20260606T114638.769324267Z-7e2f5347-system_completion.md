Learnings:
Route placement trap: a GET /auth/invite/{token} route PANICS at Mount because it conflicts with the existing GET /auth/{provider}/start — both match /auth/invite/start and Go 1.22 ServeMux finds neither more specific (one has a literal middle segment, the other a literal last segment), so it refuses to register. Moved the accept route to /invite/{token} (top-level), conflict-free. Lesson: any new /auth/<literal>/{x} route collides with the provider wildcard; keep non-OAuth routes out of /auth/.

The `authed` wrapper (sess→authMiddleware→withCSRF) is EXACTLY the invite-accept flow for free: when auth is enabled and the visitor is anonymous, authMiddleware redirects to /auth/login?next=<current URL> and the OAuth callback redirects back to ?next — so the spec's "not logged in → login then back here" needs zero custom code. But when auth is DISABLED (no OAuth providers, the localhost/test default) the wrapper is a pass-through and there's no user on the context, so the accept handler must explicitly handle currentUserID()=="" (I return 403). Don't assume `authed` guarantees a user.

Idempotency key as a context value worked so well in #085 that the same pattern shape repeats here: orchestration methods take projectIDOrSlug and resolve via GetProject — the web layer only has the slug, svc owns the slug→ID resolution, so handlers stay thin and the store stays ID-keyed.

InvitationStore mirrors MembershipStore 1:1 (central data root from cfg.DataRoot or "<DataDir>-central", per-project subdir, global .lock, WriteYAMLAtomic) — there's exactly ONE Service struct literal (service.go) and both New + NewWithEmbed funnel through it, so wiring a new central store is: add the field, construct after the others, set it in the one literal. FindByToken is an O(total-invites) walk across all project dirs because the accept route only carries the opaque token; fine at homelab scale, documented.
