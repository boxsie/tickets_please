## Learnings
- **v4 source-file directives replace `tailwind.config.js` entirely.** No more `module.exports = { content: [...], theme: {...}, safelist: [...] }`. Instead the CSS file itself owns: `@source "<glob>"` (replaces `content`), `@theme { --color-x: â€¦ }` (replaces `theme.extend.colors`), and **no safelist needed** because `@layer components` blocks are always emitted regardless of markup reference. This change alone deletes ~100 lines of safelist config that v3 needed to keep server-rendered class names like `badge-{{.Column}}` alive.
- **`@source` paths are relative to the CSS file**, not the project root. From `internal/web/static/_src/app.css` the templates dir is `../../templates`. Got this right first try, but it's the kind of thing that silently produces an empty-output build if wrong.
- **v4 has the standalone CLI on its release page**: `https://github.com/tailwindlabs/tailwindcss/releases/latest/download/tailwindcss-linux-x64` (substitute `darwin-arm64`, `windows-x64`, etc.). One binary, no Node, plays clean with the project's "single binary toolchain" ethos. The Makefile `command -v tailwindcss` pattern surfaces a clear install hint if the CLI is missing â€” same shape as the templ tooling guard from [[templ-tooling]].
- **v4 doesn't purge `@layer components`, only utility classes.** That's why the old v3 safelist (200+ classnames the templates referenced) is now unnecessary: every `.topbar`/`.badge-todo`/etc. lives in `@layer components` and ships verbatim. For UTILITY classes (`m-0`, `mt-4` etc.) introduced now in templates, v4's `@source` scan catches the literal class name string and emits the rule. Server-rendered `badge-{{.Column}}` patterns would still be a problem if you tried to use them as utilities, but they're component classes so it's a non-issue.
- **The ticket's "keep ~20 lines" hint is aspirational for a greenfield component lib, not realistic when porting a 1927-line existing stylesheet.** Took the pragmatic path: keep all existing `@layer components` content alive so the W1 page-migration tickets can swap markup to templ/utilities incrementally. The companion `@theme` tokens for badge colours and the standalone CSS variables in `:root` (legacy var(--bg) etc.) give the new tempaltes and the old templates parallel ways to consume the same palette â€” no double-source-of-truth except the four `--color-badge-*` tokens, which are intentionally duplicated against the bg+fg+border tuples in `.badge-{column}` (a single colour can't express the trio cleanly).
- **v4 minified output is ~10% larger than v3** for the same source content â€” 46k vs 42k here. Worth knowing if a future ticket budgets bytes. Likely v4's @theme machinery emits more CSS-variable scaffolding than v3's `theme(...)` resolution did.
- **The `generate:` umbrella pattern from [[templ-tooling]] composes cleanly:** added `css` to the deps list and that's it; `make build` automatically picks it up. Future generators (e.g. icon sprite, datastar runtime if it ever needs a build step) drop in the same way.
- **Idempotency check is one md5sum.** `md5sum internal/web/static/app.css` before and after `make css` matches â†’ the output is deterministic on this input. CI can promote that to a hard gate with `git diff --exit-code internal/web/static/app.css` after a fresh `make build`.
- **Inline `style=` cleanup is the smaller half of the work.** Two static escape-hatch removals + two semantic class lift-outs + two width-from-server escape-hatches that we keep. The annotated escape-hatch comments are load-bearing â€” without them, the next contributor will read the inline style as drift and try to "fix" it.
