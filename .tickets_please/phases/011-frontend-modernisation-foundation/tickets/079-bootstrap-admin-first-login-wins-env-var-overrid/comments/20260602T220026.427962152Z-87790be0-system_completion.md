Design notes worth keeping:
- env_override deliberately fires on EVERY login of the named identity, not once — because GrantMembership is idempotent (same-role grant no-ops), re-running on each login is the cheap backfill path for projects added after the admin first logged in. first_login_wins, by contrast, is naturally one-shot: isOnlyUser is only ever true for the very first account.
- isOnlyUser counts AFTER the user upsert has written the record, so "exactly one user exists and it's me" is the correct first-login predicate — don't try to pre-check emptiness before the upsert.
- The bootstrap grant sets GrantedBy = "" to mark a system (non-human) grant; the list-memberships CLI renders that as "(system)". If you add a real granter concept later, keep empty == system.
- Google's stable subject is the opaque OIDC `sub`, so the env spec matches on EITHER subject or email — operators configure `google:dan@example.com` (email) but `github:boxsie` (login==subject). bootstrapSpecMatches enforces provider equality first, then subject-or-email.
- Membership/User stores live in a CENTRAL data root, NOT the per-project .tickets_please/ tree: cfg.DataRoot or the "<DataDir>-central" sibling fallback. The recovery CLI rebuilds those stores from config to touch the exact same files the running server does — and store.New never probes the embedding provider, so the CLI works offline.
- MembershipStore.GetMembership(projectID, userID) already existed from the User+Membership store ticket (b4c3fa13) even though that ticket's learnings only enumerated List/Grant/Revoke — don't trust a learnings list as the full API surface, grep the store.
