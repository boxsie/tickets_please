## Work summary
Defined `vecindex.Sidecar{Provider, Model string; Dim int; Vec []float32}`. `WriteSidecar(path, sc Sidecar)` and `ReadSidecar(path) (Sidecar, error)` now operate on the struct — no back-compat reader for the flat-array form. Added `model` field + `SetModel(string)` setter on `worker.Worker`. Worker.process now stamps `Provider: w.provider.Name()`, `Model: w.model`, `Dim: len(vec)`, `Vec: vec` into every sidecar. `Service.hydrateMount` calls `s.Worker.SetModel(s.Cfg.OllamaModel)` (idempotent) so the worker has the model name before any writes for that mount. Hydrate's read path uses `sc.Vec` for the index Upsert; sc.Provider/sc.Model are parsed but ignored (W2-T3 will use them for staleness).

## Learnings
Provider/model identity flows through the worker by setter rather than constructor: `worker.Worker.SetModel(string)` is called from `Service.hydrateMount`. The constructor route would have been cleaner but `worker.New` is wired in `service.go` which was off-limits for this ticket — `SetModel` is the workaround. When future work lands per-project providers (W2-T1), `SetModel` should be called *per mount* (not just at hydrate) since different projects can run different models. Also: sidecar `Dim` is redundant with `len(Vec)` but keeping it explicit lets staleness logic in W2-T3 cheaply check "was this written by a 1024-dim model" without examining the slice. Merge with W1-T1 hit a hydrate.go conflict — both tickets touched the same dim-check block; resolved by combining `len(sc.Vec) == s.EmbedDim`.
