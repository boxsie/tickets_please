Learnings:
Pre-existing race surfaced by per-mount workers: AssignTicketToPhase had no Worker.Flush before BeginOp, so an in-flight body-sidecar write (Stat→Embed→WriteSidecar) could recreate the ticket dir mid-RenameDir. Fixed by adding mount.Worker.Flush(ctx) before BeginOp, mirroring DeleteX. The "Worker.Flush before destructive ops" learning generalizes from "remove a tree" to "any rename or remove a tree". Goroutine-leak tripwire: freshServiceWithCfg used cfg.DataDir = t.TempDir() (no .tickets_please suffix), which silently no-op'd RegisterProjectMount in createProjectImpl's post-create — old tests ran with no mount and the embed pipeline went through s.defaultIndexes. Switching to per-mount workers required tests to actually have a mount, so changed freshServiceWithCfg to filepath.Join(t.TempDir(), ".tickets_please"). Watch for similar silent-no-op patterns elsewhere. The W1-T2 note "for per-mount providers, SetModel should be called per mount" turned out wrong: with per-mount workers, SetModel itself becomes dead weight — model name is intrinsic to the worker via constructor, no setter needed. Stop vs Flush semantics matter: Stop is destructive (cancel + drain + close); Flush is a barrier (sentinel through queue, queue stays open). Keep them distinct.
