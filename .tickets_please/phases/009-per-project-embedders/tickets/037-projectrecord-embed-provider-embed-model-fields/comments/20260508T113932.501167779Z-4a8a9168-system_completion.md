✅ Ticket completed.

Testing evidence:
`make build` clean; `go test ./...` green. `internal/svc/projects_test.go` now includes: stamps ollama+model on create, openai default model on create, update writes/round-trips embed fields, and a legacy-yaml-without-embed-fields fixture that loads cleanly via `omitempty` and gets backfilled on next update. Round-trip through yaml confirms both fields appear in fresh `.tickets_please/project.yaml` files.

Work summary:
Extended `store.ProjectRecord` with `EmbedProvider` and `EmbedModel` fields, both `omitempty` in YAML so legacy project.yaml files load without changes. New `defaultEmbedFor(cfg)` helper in `internal/svc/projects.go` picks provider from `cfg.EmbedProvider` (defaults to "ollama") and pairs it with `cfg.OllamaModel` for ollama or hardcoded `text-embedding-3-small` for openai. `CreateProject` stamps both fields into the record at creation. `UpdateProject` writes them when the input pointers are non-nil (trim-spaced); a no-op input leaves them untouched. Extended `domain.UpdateProjectInput` with `EmbedProvider *string` / `EmbedModel *string` so form/MCP layers can wire through. Web handler + MCP tools were not modified — that's a future ticket once the Settings page lands.

Learnings:
The `*string` pointer pattern in `UpdateProjectInput` (vs plain `string`) lets the update layer distinguish "field omitted from form" from "field set to empty string". Use this whenever a form field has a meaningful "leave alone" semantic. Also: hardcoded OpenAI default model name (`text-embedding-3-small`) lives in `defaultEmbedFor` rather than config because OpenAI doesn't have a server-wide default knob in this codebase — if/when an `OpenAIModel` config field gets added, point `defaultEmbedFor` at it. The `omitempty` tag is the entire backwards-compat story for project.yaml — no migration script needed; legacy yamls just don't emit the fields, and on next save they pick up the values from `defaultEmbedFor`.
