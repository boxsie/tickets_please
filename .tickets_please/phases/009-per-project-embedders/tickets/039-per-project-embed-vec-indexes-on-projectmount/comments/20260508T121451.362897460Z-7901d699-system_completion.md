Testing evidence:
`make build` clean; `go test ./...` green across all 11 packages on merged main. `go test -race ./internal/svc/ ./internal/worker/` also green — race detector caught (and the agent fixed) a workerIndexResolver lock-scope issue mid-development. New `internal/svc/service_per_project_test.go` covers: two fake providers (768d + 1024d), two project yamls with different embed_models, both mounted, asserts per-mount EmbedDim, distinct *vecindex.Index pointers, distinct providers, correct mountProviderAndIndex routing, and that searching one mount returns only that mount's hits. Eviction test confirms Store + per-mount indexes get nilled at LRU cap.

Work summary:
Refactored `embed.New` to take a small `EmbedConfig{Provider, Model, OllamaURL, OpenAIKey}` view instead of full `*config.Config` — decouples the embed package from server config and lets per-mount build merge project.yaml overrides over server defaults. `ProjectMount` gained `Embed`, `EmbedDim`, and the four `*vecindex.Index` fields; `Service` dropped its four `*Idx` fields. New `attachMountEmbedAssets` builds per-mount provider via `Service.EmbedNew`, probes it, allocates indexes at the probed dim. Eviction nils per-mount indexes (matching the existing Store-nil pattern). Search routing: project-scoped paths go through `mountProviderAndIndex(slug, kind)` with fallback to a Service-level `defaultIndexes` quartet; cross-mount paths walk mounts, embed once per provider (since dims may differ), merge by score. Worker stayed global with a tiny `IndexResolver` callback so writes land in the right per-mount index based on `Job.Owner` — per-mount worker is W2-T2's job. `Service.EmbedNew` is the test-injection seam; `NewWithEmbed` wires it to a closure returning the supplied fake regardless of view, so unit tests can override per-mount dims directly.

Learnings:
Test-injection seam matters: `Service.EmbedNew func(EmbedConfig) (embed.Provider, error)` (defaults to `embed.New`, tests override) lets per-mount build run in unit tests without dialing out. Cross-mount search with mixed dims has a real cost — when two mounts run different models, you can't embed the query once and search both indexes (wrong dim for one), so the implementation embeds per-provider and merges by score. Cache-by-provider-fingerprint is premature at hobby scale. The W1-T2 "SetModel per mount" guidance was half-right: with a still-global worker in W2-T1, SetModel lives in `hydrateMount` but that means whichever project mounted most recently sets the model for everyone — works for stdio (one project), racy for HTTP. W2-T2 lands per-mount workers and fixes this. mountsMu lock contract: per-mount index reads must hold the lock for the full read, since indexes get nil'd on eviction (race detector caught a first-attempt bug here). Method-value idiom blocks accessor methods: tests passed `s.SummaryIdx.Len` as a `func() int`, so converting `SummaryIdx` to a method returning `*Index` breaks tests; use test-only helpers in `_test.go` instead.
