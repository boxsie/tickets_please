Made create_project auth-soft — the bootstrap escape valve that breaks the chicken-and-egg between create_project and register_agent. Added svc.optionalSession (returns nil agent on no-session ctx instead of erroring). Switched svc.CreateProject from requireSession to optionalSession; guarded CreatedByAgentID and domain.Project.CreatedBy nil-paths; StageOp.Commit was already nil-agent safe (skips the auto-commit). At the MCP layer, handleCreateProject no longer routes through callWithRetry — it threads the agent ID into ctx if a session exists for attribution, otherwise calls svc.CreateProject directly. Learnings:
The auth-on-create-project gate was load-bearing for two-step bootstrap: it forced the LLM to register first, which in turn forced project.yaml to exist, which forced... CreateProject. Dropping the gate is the right cut because attribution genuinely doesn't apply to "the very first record being created." Created_by has been nullable in the schema all along (CreatedByAgentID is *string, AgentRef is *domain.AgentRef) — that hint should have caught my eye earlier; the schema knew this was the right shape before the auth check did. StageOp.Commit at internal/store/stage.go:188 already had `if o.store.AutoCommit && agent != nil && len(touched) > 0` so the bootstrap path's auto-commit suppression is automatic — no extra guard needed in CreateProject. The MCP-layer trick of "skip callWithRetry but still thread the agent ID if one exists" preserves attribution when a session happens to be registered (e.g. stdio's pre-registered session) without forcing it. TestCreateProject_RequiresSession was the canary that caught the breaking change — every removed-requirement should expect to break a "Requires" test. TestIntegration_AgentSessionExpiry's switch from CreateProject to CreateTicket is the load-bearing pattern: any test that asserts ErrUnauthenticated on a mutation must use a *required-auth* mutation; CreateProject is no longer one. Earlier docs-only fix was the wrong scope — the real lesson is "if the recommended workaround in your error message requires tools the user doesn't have, you haven't fixed the bug, you've just renamed it."</learnings>
