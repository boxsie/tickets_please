Learnings:
CreateProject did not previously go through RegisterProjectMount — it wrote to s.Store directly and relied on the s.Store fallback inside ResolveProjectStore for in-process resolution. That worked but bypassed any persistent mount tracking. Discovered while writing the DeleteProject test: registry.yaml stayed empty after CreateProject succeeded. Fix: call s.RegisterProjectMount(repoPath) at the end of CreateProject when DataDir follows the convention (filepath.Base equals .tickets_please). This is the right shape architecturally too — every successful create or load should leave the same mount footprint. The convention check matters because tests use a bare tempdir as DataDir to skip the eager-mount path; without the check the self-register would error against the wrong parent. ProjectMount.RepoPath is the absolute repo path (parent of .tickets_please), not the data dir itself — when reading mount.RepoPath in DeleteProject, that is exactly what gets passed to removeFromMountRegistry. The atomic-write pattern in store.WriteYAMLAtomic creates a .tmp-* file in the same dir then renames. The defer-Remove on tmpName is best-effort, so the post-rename cleanup might leave nothing because the rename succeeded. The test that asserts no leftover .tmp files catches a regression where the cleanup logic flips. Boot-time mount errors are logged at WARN not ERROR — they are expected (user might have deleted a repo since the last boot) and the service must come up regardless. A panic or fatal would be a bug. Holding mountsMu across the registry I/O is fine because the registry write is fast (sub-ms small YAML) and not in any hot path.
