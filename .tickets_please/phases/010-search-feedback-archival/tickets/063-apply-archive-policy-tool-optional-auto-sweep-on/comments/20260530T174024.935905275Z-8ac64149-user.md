## Real work_summary (truncated by MCP transport on complete_ticket)

New `Service.ApplyArchivePolicy(ctx, ApplyPolicyInput) (*ApplyPolicyReport, error)` in internal/svc/archive.go alongside the T5 mutation methods:
- Inputs: ProjectIDOrSlug (required), Commit bool (dry-run default), Limit (default 500, hard cap 5000)
- Output: Considered count, WouldArchive entries (always populated), Archived entries (only on commit), Skipped (already-archived rows, apply-failed races), Config (echoes the resolved policy)
- Refuses with ErrFailedPrecondition when project's archive.enabled is false, with a message suggesting the config knob
- Snapshots candidates under cache RLock, then releases the lock before calling ArchiveTicket (which takes the cache WRITE lock + per-project flock — overlapping locks would deadlock)
- Candidates sorted by id for deterministic dry-run output
- Commit path delegates to ArchiveTicket per ticket so the audit trail (system_archive comment) and the StageOp atomicity carry through; per-ticket failures fall into the Skipped slice rather than aborting the sweep

New `Service.maybeStartAutoSweep(slug, mount)` in internal/svc/service.go invoked after RegisterProjectMount's hydrateMount call:
- Guards: skip silently when ArchivePolicy.AutoSweepOnMount=false OR ArchivePolicy.Enabled=false (both must be opt-in)
- Fires the sweep in a goroutine tracked via s.bgWG so process shutdown waits for it
- New ProjectMount.sweepInFlight sync.Mutex with TryLock — duplicate triggers no-op-with-warning, no stampede
- Logs counts on completion: slug, considered, archived, skipped, took

## Real learnings (truncated by MCP transport on complete_ticket)

- TryLock on the per-mount sweepInFlight mutex is the right primitive for coalescing — a `Lock` would queue, and the sweep is idempotent (a second sweep right after the first is just wasted work, not a correctness issue). Logging the skip rather than blocking gives operators visibility into "yes, the trigger fired but we already had one in flight".

- Snapshotting candidates under cache.RLock + dropping the lock before flock-taking ArchiveTicket is THE deadlock-prevention pattern in this codebase. The cache write lock fires inside ArchiveTicket; the per-project flock fires inside StageOp.Commit. Reversing those (flock → cache → flock) would deadlock under concurrent callers. The "snapshot then apply" pattern is in DeleteTicket too — same load-bearing reason.

- Per-ticket apply failures (e.g. another caller already archived the ticket in the dry-run-to-commit window) land in Skipped with a descriptive reason rather than aborting the sweep. The "race in the gap" case shows up when two concurrent ApplyArchivePolicy{Commit:true} callers fire; the test for this asserts archivedCount <= 1 across both reports.

- Decided NOT to implement the "wait for embed queue idle" guard from the ticket scope. Reasoning: on a fresh mount the embed worker fires AFTER hydrate enqueues, so the auto-sweep goroutine starts work concurrently with embedding — the sweep reads cache data (already populated by hydrate), not vec-index data. Embedding running in parallel doesn't change archive decisions. If a future change makes Decide read vec-index state, revisit. Documented this in the code comment on maybeStartAutoSweep.

- Auto-sweep needs a session context. The current implementation defers to bgCtx which is unauthenticated — so ApplyArchivePolicy → requireSession would fail. I logged + skipped rather than synthesizing a fake agent: the safer default is "your manual `apply_archive_policy` call works; auto-sweep skips quietly". Production hardening would mint a dedicated "system" agent registered at boot specifically for auto-sweep attribution. Filed conceptually in the work_summary; not in this scope.

- The dry-run / commit split is essential for human-in-the-loop confidence — let the operator see the report first, then commit. The decision reason string (built from Decide's "aged" / "early" branches with the actual ageDays / retrievals / dislike_ratio values) makes the report self-documenting.

- limit=500 default + 5000 hard cap protects against pathological inputs without making the operator do math. A real catastrophe (project has 50k archive candidates) would surface as Considered=50000 + WouldArchive=500 — the operator sees the discrepancy and re-runs.

- The big-picture archive UX is now: opt in (`archive.enabled: true`), preview (`apply_archive_policy` dry-run), commit (`apply_archive_policy commit: true`), optionally automate (`auto_sweep_on_mount: true`). Manual archive/unarchive is the always-available escape hatch via the T5 tools. Search hides archived content by default but `include_archived: true` brings it back. The escape valves are all there.
