## Real learnings (truncated by MCP transport on complete_ticket)

- Flat-layout WithProjectLock resolves to <Root>/.lock — same flock every StageOp commit takes. Feedback mutations serialize against ticket writes; fine for now but worth profiling once T3's RecordRetrieval fires on every search hit.

- yaml.v3 needs the on-disk map keyed by plain string (not the EntryKey defined-type) to avoid a custom MarshalYAML — keep the on-disk feedbackEntryRecord shape distinct from domain.FeedbackRecord for that conversion.

- Treating Version==0 as v1 keeps hand-edited feedback.yaml usable for grep-and-debug while still tripping the guard on a real future v2 file.

- FOUR mount lifecycle paths need attachMountFeedback: (1) fresh RegisterProjectMount, (2) idempotent re-register re-mounting an evicted entry, (3) stdio-fallback inside ResolveProjectStore when registry empty but s.Store hosts the slug, (4) re-mount inside ResolveProjectStore's mount.Store==nil branch. The grep test: rg attachMountEmbedAssets internal/svc/ — every call site should be paired with attachMountFeedback (except rebuildMountEmbedAssets in ReembedProject, since Feedback survives an embedder swap).

- Eviction must nil mount.Feedback alongside mount.Store for symmetry — today no actual leak (FeedbackStore holds no fds) but the post-eviction Store would otherwise be reachable through Feedback's back-reference.

- DeleteTicket's feedback cascade fires while lp.Lock is still held (deferred Unlock at function exit) — no deadlock today because nothing in the flock path also takes lp.Lock, but if a future change adds a flock-takes-lp.Lock path this will deadlock; move the DeleteMany outside the deferred Unlock window or fire from a separate goroutine.

- gopls flags cross-package refs in this dual-mount workspace as undefined because /mnt/data and /home/dan both point at the same files but gopls only indexes one path — trust go build / go test results, not the red squiggles.
