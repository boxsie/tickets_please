## Real work_summary (truncated by MCP transport on complete_ticket)

Added `EntryKey domain.EntryKey` field to all three search hit types (TicketHit, CommentHit, LearningHit) in internal/svc/search.go. Each Search method populates the field via the T1 helpers (TicketEntryKey, CommentEntryKey, LearningEntryKey) at result construction time.

Added `recordSearchRetrievals(ctx, map[slug][]EntryKey)` helper on Service that batches one per-mount feedback.RecordRetrieval write per call. Failures are logged but never propagated — search result is the user-facing artifact; degrading the W3 archive signal beats failing the search.

- SearchTickets: restructured the result loop into an immediately-invoked anonymous function so the lp.Lock RUnlock is deterministic before the flock-taking feedback write (deadlock prevention).
- SearchComments: both the project-scoped path and the cross-mount aggregate path build a per-slug entry-key map and call recordSearchRetrievals before returning.
- SearchLearnings: same per-slug map pattern on the cross-mount aggregate path.

Added entry_key field to every formatted search hit in internal/mcptools/format.go (formatTicketHit / formatCommentHit / formatLearningHit). New feedbackHint([]string) helper returns nil for empty key sets and otherwise the canonical {tool, entry_keys, note} block.

Each of the three search handlers collects entry keys in result order and attaches feedback_hint to the response body only when at least one hit came back. No nag on empty searches.

Per the design: did NOT add per-hit feedback counters (would encourage gaming and bloats responses; counts are observable via feedback.yaml when debugging).

## Real learnings (truncated by MCP transport on complete_ticket)

- The deferred-RUnlock + re-RLock dance I wrote first for SearchTickets was ugly; replacing it with an immediately-invoked anonymous function gives a clean scope where the lock is bounded and the post-lock feedback write is the last thing the function does. Right shape for any "hold cache lock, build result, then do flock-taking I/O" pattern.

- Cross-mount search paths (SearchComments unscoped, SearchLearnings) need a map[slug][]EntryKey accumulator because feedback writes are per-project. The slug is already in the scored.slug field (or recovered via ownerByID for the stdio fallback).

- The empty-hits skip on feedback_hint matters — without it, an LLM that searches for something that doesn't exist gets nagged to "rate these (none)", which is just noise. Same logic at the svc layer: recordSearchRetrievals no-ops on empty per-slug maps so we don't write to feedback.yaml just to record "I searched for nothing".

- feedback_hint belongs at the top level of the search response (sibling of hits), not inside each hit. Putting it per-hit would suggest "rate this one in particular" — but the LLM already has all the entry keys via hit-level entry_key, and the top-level hint reinforces "here's the bulk rate tool, here's the keys you can pass to it."

- The test for empty-feedback-on-zero-hits had to live at the svc layer (TestSearchTickets_EmptyResultsNoRetrievalWrite) — the MCP-layer test couldn't easily produce a zero-hit search because the fakeEmbed implementation always ranks SOMETHING first.

- The fakeEmbed-based handler test needed a waitUntilSearchHits poll helper because the embed worker fires asynchronously after CreateTicket. ~2s budget (40 × 50ms) is comfortable.
