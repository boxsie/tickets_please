Learnings:
SearchTickets requires a project filter at the service layer for v1 (rejects empty ProjectIDOrSlug with explicit message about cache thrashing). The web handler should catch this earlier and surface a UX-shaped error pointing the user at the slug filter, since the service-level message leaks implementation detail. Capturing search errors into the page-data struct as body.Err is better UX than returning a 500 — typo in slug, transient embed failure, etc all render inline next to an empty results block. Using one results partial that branches on kind via if-else-if is verbose but keeps a single hx-target dead simple; a per-kind partial would push branching into the page template instead, which is worse. The list template func is a small but disproportionately useful idiom for tab strips and similar enumerated UIs — beats threading a Go-side []string through page data when the values are constants. SearchLearnings and SearchComments both accept empty ProjectIDOrSlug as cross-project, but SearchTickets does not — this asymmetry is intentional per the service layer comments and shouldn't be papered over in the web tier. Score values from fakeEmbedder are deterministic but uninteresting; tests that assert on structural shape (search-hits or No matches) are more robust than tests that compare numeric scores.
