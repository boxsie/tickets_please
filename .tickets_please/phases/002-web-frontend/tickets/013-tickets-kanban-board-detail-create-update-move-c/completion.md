## Work summary
Added internal/web/handlers_tickets.go with eight handlers: handleBoard, handleTicketNewForm, handleTicketCreate, handleTicketDetail, handleTicketEditForm, handleTicketUpdate, handleTicketMove, handleTicketComplete. ## Learnings
Service.GetTicket walks hostStoreForTicket to find the project — there's no slug-direct entry point yet. The ?slug= query hint convention is purely a UX optimisation reserved for a future svc.GetTicketInProject(slug, id) variant; today it's forwarded through templates to keep URLs stable but the handler doesn't consume it. The move-to-done short-circuit at the handler layer is worth keeping even though Service rejects too — the inline error message can point at the Complete form whereas Service's generic error is opaque. Comma-separated form inputs for depends_on plus parallelizable_with are fine for v1; collectIDs accepts both shapes so a future htmx type-ahead picker can swap the input type without touching handlers. The ticket detail template needs three template helpers (deref for *string, phaseSlug to look up by ticket.PhaseID, mkAssignPhase to package the partial payload) — the alternative was building everything in the handler, which would have made the data type fat and untestable. The funcMap pattern keeps templates declarative. Goldmark fenced code rendering is handled cleanly by the existing markdown helper; no extra config needed for ticket bodies.
