## Learnings
WaveSection is shared by phase-detail AND the index (inside each phase's <details>), so per-wave deep-link affordances can't be unconditional: bare id="w3" would collide across phases on the index. Gated both the anchor id and the "Focus on this wave →" link behind a Focusable bool on WaveSectionProps — true on detail (one phase owns the page), false on index. Key elegance: the focus link is a query-only relative href "?wave=N", so it resolves against whatever URL it's on — focuses one wave on detail, the same wave across all phases on the index — no slug threading needed. The ?wave=0 (unassigned) wave means presence must be tracked separately from the zero value, hence parseWaveParam returns (int, bool) via q.Has("wave"). The SSE phase re-render (renderPhasePatch) shares the #phase-waves-{id} morph target with both pages, so it must use the focusable wave props too or a live ticket event silently strips the detail page's anchors/focus links until reload; accepted the minor cost that index live-updated phases then also get focusable markup (harmless: relative focus links valid there, bare ids only nominally dup across same-session live-updated phases). Scroll+highlight needs ZERO JS: html{scroll-behavior:smooth} + .phase-wave:target{animation pulse} does it natively. The only JS (static/phases.js) is localStorage collapse persistence, keyed tp:phase-open:{project_id}:{phase_id} (project-scoped per the hint so same-slug phases across projects don't collide) — and it MUST bail in focus mode (?wave/?phase in URL) or it persists the server's transient force-open state into normal browsing. Added data-phase-id + data-project-id data-attrs as the JS hooks. Gotcha: an existing test pinned the literal `<details class="phase-row">` string — adding data-phase-id + open?={} broke the exact match; assert on `data-phase-id=` presence and absence of ` open><summary` for the collapsed check instead. go:embed static auto-recurses so phases.js shipped with no registration.
