Learnings:
- The vec-index resident layout makes per-id deletion straightforward: `Index.Delete(id)` is the inverse of `Upsert` and `RemoveByOwner` is for project-scoped wipes. For DeleteTicket I needed both shapes — Delete for the ticket body and learnings entries (keyed by ticket UUID), and a small loop calling Delete for each cached comment ID in `lp.Comments[id]`. No need for a new "RemoveByTicket" helper since the cached comment slice is the canonical comment-ID source.

- `findTicketDir` already returns the relative path that StageOp.RemovePath wants, including the phased-vs-phaseless prefix (`tickets/<NNN>-…` or `phases/<NNN>-…/tickets/<NNN>-…`). Reuse it everywhere instead of reconstructing — phases/phaseless drift would silently break the delete on phased tickets.

- Worker.Flush(ctx) before BeginOp is the load-bearing step that prior delete paths (DeleteProject, DeletePhase) all do. Without it, an in-flight embed job for the doomed ticket can write a sidecar back into the dir mid-Remove, leaving a non-empty residual directory. Match that pattern verbatim for any future "remove a tree" Service method.

- `ListTools()` returns a map, so the canonical-list test uses a sorted []string projection. When adding a tool, three places need the same number bump in lockstep: `cmd/tickets_please/main.go:totalTools`, `internal/mcptools/tools.go` doc comment ("registers all N tools"), and `internal/mcptools/tools_test.go:expectedTools`. The test fails loud and prints the actual list which made the drift trivially diagnoseable.

- Native `<dialog>` modals with a small inline script remain the right primitive for confirm dialogs in this no-JS-framework codebase (continues the pattern from ticket 031). The Cancel button needed a `data-dialog-close` partner for `data-dialog`; I added a generic close-handler in the same script so future dialogs get it for free without duplicating per-dialog wiring.

- Refusing on dependents needs to scan every other ticket's DependsOn under `lp.Lock.Lock` (not RLock) so we don't race a concurrent CreateTicket adding a new dep mid-check. The scan is O(active tickets × deps-per-ticket) which is fine at this project's scale; if it ever matters, an inverted index (ticket id → dependents) on LoadedProject would close it, but premature.

- The web smoke tests pass `?slug=` on every ticket-mutation URL because the handler uses it for redirect-back hostStoreForTicket avoidance. New web ticket-mutation routes should keep accepting and forwarding it; the new delete handler matches.
