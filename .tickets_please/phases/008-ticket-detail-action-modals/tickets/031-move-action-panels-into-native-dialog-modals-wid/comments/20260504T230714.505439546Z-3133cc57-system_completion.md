Learnings:
Native `<dialog>` is the right modal primitive in this codebase for the same reason `<details>` won the dropdown bake-off (past learning, ticket 89efc4cf): no framework needed, browser handles backdrop + focus trap + ESC, click-outside-close is one event listener (`if (e.target === dlg) dlg.close()`). The close button is a `<form method="dialog">` — that's the native pattern, no JS at all for close. The trigger pattern is also tiny: `[data-dialog="dlg-id"]` attributes scanned once at script-init, calling `dlg.showModal()` on click. Total inline JS came in at ~12 lines. `dialog.showModal()` (modal) is what you want vs `dialog.show()` (non-modal) — only the former gets the `::backdrop` pseudo and focus trap.

Existing form partials (`move_form.tmpl`, `complete_form.tmpl`, `assign_phase_form.tmpl`) drop into a modal body without modification because they use `class="stack"` plus standard inputs — no chrome was load-bearing for the sidebar layout. The `.ticket-side form.stack` background-reset rule was needed because `form.stack` has a default elevated background; the modal needed the same reset (`background: transparent; border: none; padding: 0`). I added an equivalent `.modal-body form.stack` rule so future modals hosting other `stack` forms render cleanly without per-modal overrides.

The two render branches in detail.tmpl (frozen vs active) had drifted — frozen was wrapped in `.ticket-grid` even though it only used `.ticket-main`, an artifact from when both branches symmetrically had a side rail. Collapsing to a single `.ticket-main` wrapper for both branches halves the template duplication and means `.ticket-grid` could be removed from CSS + safelist entirely.

Validation-failure UX is the only thing that regresses slightly: today a failed Complete submit re-renders the page, which means the dialog comes back closed and the user has to reopen it to see the inline error. Acceptable for v1 — the existing flash/error rendering is still on the page — but the natural follow-up is to teach the form to htmx-swap into `.modal-body` so failures stay inside the dialog.

`tailwindcss@3.4.10` regen is fast (~600ms) and safelisting works exactly as documented; the new `modal-*` classes appear in the minified output once added to `safelist`. The compiled `app.css` grew by ~600 bytes for the modal block.

Couldn't interactively click the dialog open/close in a browser from this session (no browser-automation tool available); leaned on curl + markup inspection + standard `<dialog>` semantics. Future tickets touching modal interaction should ideally be smoke-tested in a real browser before completion.
