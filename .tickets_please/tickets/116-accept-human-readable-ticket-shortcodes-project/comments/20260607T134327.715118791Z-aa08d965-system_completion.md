Learnings:
Shortcode resolution landed as svc.ResolveTicketRef(ctx, defaultSlug, ref) + a tiny tools-layer wrapper, wired into EVERY ticket-targeting handler at the boundary (get/update/move/complete/add_comment/assign/archive/unarchive/delete/list_comments + list_comments_scoped's optional ticket_id). Key facts for future work: (1) domain.Ticket has NO Number field — the global number lives only in store.TicketRecord.Number and the on-disk dir name `<NNN>-<slug>`. So number→UUID resolution must walk the host store's ticket records (st.WalkTickets), it can't read the cache (lp.Tickets is keyed by UUID with no number). (2) Classification is by SHAPE, not by validating UUID format: parseTicketShortcode treats only `<all-digits>` (bare number) or `<slug>/<all-digits>` as shortcodes; everything else (UUIDs, opaque ids) passes through untouched — so it can never misfire on a real UUID. Split on the LAST '/' since slugs are single-segment URL-safe (no '/'). (3) Bare numbers need the session's bound project; added tools.defaultProjectSlug — the LENIENT sibling of resolveProjectSlug (returns "" instead of erroring) because a UUID needs no project context, only a bare number does. resolveProjectSlug uses req.GetString (nil on raw envelope) but defaultProjectSlug uses decodeArgEnvelope so it stays envelope-robust. (4) To get a store from a slug: s.ResolveProjectStore(ctx, slug) (re-mounts from RepoPath if LRU-evicted; falls back to the default stdio store). (5) Resolution is a pure passthrough for UUIDs — no existence check at resolve time; downstream GetTicket/etc. still validate. Did NOT surface the number in API output (formatTicket has no number) — agents already see numbers from commit messages/conversation per the ticket framing; surfacing it would need plumbing Number through domain.Ticket+hydrate, out of scope.
