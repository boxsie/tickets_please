Learnings:
The web sign-in window is the tp_user login cookie in internal/web/auth.go, NOT tp_sid. Two distinct web cookies: (1) tp_sid (session.go) = the per-request AGENT identity, 7-day, but resolve() transparently MINTS A FRESH agent when the record is missing/expired — so tp_sid expiry never logs the human out, it just rotates the web-ui agent id. (2) tp_user (auth.go) = the actual login; validated by HMAC signature only (no embedded timestamp), so its lifetime is purely the cookie MaxAge. The signing secret is derived from cfg.DataRoot (session.go newSessionManager), so it's STABLE across restarts — rebuilding/restarting the binary does NOT invalidate logins (I'd suspected that as the 'too short' cause; it isn't). Was hardcoded 30 days; now config-driven via auth.session_max_age_hours: >0 = hours, 0/unset = indefinite (default, per the user's 'indefinite tbh'). Gotcha: http.Cookie MaxAge==0 means a SESSION cookie (dies on browser close) — the opposite of indefinite — so 'indefinite' is encoded as a ~10-year finite MaxAge (indefiniteCookieMaxAge const), never literal 0. Config plumbing: added SessionMaxAgeHours to config.AuthConfig (koanf:"session_max_age_hours") + a defaults entry "auth.session_max_age_hours":0 (dotted key → nested under auth; not env-overridable since env var names can't contain '.', which matches the 'auth is file-only' design). examples/config.yaml had NO auth: block at all — added a documented one. Left the MCP agent-session TTL (agent_session_ttl_minutes) and tp_sid untouched per scope.
