Learnings:
UUID-prefix resolution is a pure backstop bolted onto the EXISTING shape-classification in svc.ResolveTicketRef (tickets.go) — it did NOT touch parseTicketShortcode (that still returns isShortcode=false for hex prefixes; tests depend on its contract). Flow: parseTicketShortcode handles number/slug-number first (priority), and only in the `!isShortcode` branch (formerly pure passthrough) do we try resolveTicketPrefix before falling back to passthrough.

Key design choices that matter:
- Guard is isUUIDPrefix: pure hex [0-9a-fA-F], len 4..31, REQUIRES at least one a–f letter. The letter requirement is load-bearing: a pure-digit ref like "1234" must stay a number-shortcode (handled earlier), and a full dashed UUID (36 chars, has '-') fails both charset and length so real UUIDs/opaque ids still pass through untouched — never hit the walk. So the change is purely additive; every existing passthrough/number/slug-number case is unchanged.
- Scope: bare prefix → defaultSlug (session-bound project); "<slug>/<prefix>" → named project (re-split on last '/' inside resolveTicketPrefix since parseTicketShortcode already rejected it).
- Outcomes: 0 matches → fall back to passthrough (NOT an error) so genuine opaque ids still reach the downstream existence check; exactly 1 → resolve; >1 → ErrInvalidArgument naming prefix+project. Unknown/unbound project (ResolveProjectStore errors) → passthrough, not a hard error.

TESTING GOTCHA: you cannot exercise the ambiguous (>1 match) path with real CreateTicket'd tickets — random UUIDs effectively never share a 4+ hex prefix. I extracted matchPrefix(ids, prefix) as a pure helper precisely so ambiguity is testable with crafted ids (TestMatchPrefix), and added TestIsUUIDPrefix for the guard. The full-path integration test in resolve_ticketref_test.go only covers unique-match + slug/prefix + unmatched-passthrough.
