Learnings:
Confirmed the complaint was the SET path (MCP arg-decode envelope), not the web render path — exactly the same root cause as complete_ticket (5688ace3). Ported create_ticket, update_ticket, add_comment, move_ticket, AND assign_ticket_to_phase off req.RequireString/req.GetString/req.GetArguments onto the envelope-robust requireStringArgs + decodeArgEnvelope helpers already in tools.go. The trap: it's not just RequireString — req.GetString("body","") and req.GetArguments() BOTH read via GetArguments() too, so on a json.RawMessage envelope the body silently came back "" and optional fields (phase/wave/depends_on) silently dropped, not just the required ones erroring. So when porting a handler you must replace ALL three call shapes in it, then read optional fields off the decodeArgEnvelope map. decodeArgEnvelope returns one map you can reuse for every optional field in the handler. Remaining RequireString callers that are NOT free-text (project/phase slugs, target_column enum, query, model/client_name in register_agent) were left as-is; resolveProjectSlug still uses req.GetString but is safe because it falls back to the session-bound slug. Regression test pattern: build mcp.CallToolRequest with Params.Arguments = json.RawMessage(json.Marshal(args)) — a plain map[string]any never exercises the bug because that's the fast path. New tests live in internal/mcptools/richbody_envelope_test.go; they reuse seedTicketForCompletion/callRegister/freshToolsForRegister from the existing test files.
