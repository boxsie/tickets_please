Work summary:
Root: `embed.Ollama.Probe` auto-pulled a missing model synchronously (up to 15m, 90s ctx), and `attachMountEmbedAssets` runs inside `svc.New` (eager-mount + registry-restore) before `ServeStdio` — so a missing per-project model blocked the boot path past the 30s MCP handshake.

`internal/embed/ollama.go`:
- `Probe` no longer pulls — fast Embed("ping") that records dim or returns a model-missing error.
- Added `EnsureModel(ctx)` (probe-then-pull-if-missing; no-op if present) and exported `IsModelMissing(err)` + `const ModelPullTimeout`.

`internal/embed/provider.go`: added optional `ModelEnsurer` interface (implemented by Ollama; OpenAI deliberately doesn't).

`internal/svc/service.go`:
- `newServiceCore` server-default probe: on model-missing + ModelEnsurer, pulls once synchronously (preserves fresh-deploy usability for the one model we must have; documented as the one-time install cost). Non-pullable/other errors stay fatal.
- `attachMountEmbedAssets` fallback branch: falls back to server default immediately, stamps the fallback's REAL model (truth — fixes de1a552e), and if the failure was model-missing, calls `startEnsureMountModel` (background goroutine: pull → rebuildMountEmbedAssets swap → flush/wipe-sidecars/hydrate re-embed). Other failures = permanent fallback.
- New `startEnsureMountModel`/`ensureMountModelAsync`; added `bgCtx`+`bgWG` fields; `Close` cancels (via backfillCancel) and `bgWG.Wait()`s before stopping workers.

`internal/svc/hydrate.go`: `staleSidecar` re-gained the dim check (de1a552e).

Learnings:
- The boot blocker wasn't only the per-mount probe: `svc.New`→`newServiceCore` ALSO probes the server-default provider synchronously (90s ctx), and mounts attach INSIDE `svc.New` (eager-mount of cfg.DataDir + mount-registry restore at lines ~256-291) BEFORE `cmd/main.runMCP` calls `ServeStdio`. So anything that pulls during attach blocks the handshake. Keep mount attach off the synchronous boot path or guaranteed fast.
- Splitting "probe" (fast, dim only) from "ensure" (pull) via a separate `ModelEnsurer` interface is cleaner than a bool flag: OpenAI simply doesn't implement it, so the service's type-assert naturally means "nothing to pull → permanent fallback." `IsModelMissing` must match on the wrapped error string ("status 404"+"not found"+"try pulling it"), and buildMountProvider wraps the probe error with %w, so string-contains still works through the wrap.
- The fallback truth-stamp (de1a552e) and the non-blocking pull (this ticket) are the SAME code branch — do them together. Stamp `embedViewFromCfg(cfg,"","").Model` (the server default's real model), not the requested model.
- Background goroutine lifecycle: reuse the boot-backfill context (cancelled by backfillCancel) and a WaitGroup; `Close` must `bgWG.Wait()` BEFORE stopping per-mount workers, because the goroutine's swap path stops/realloc's the mount's worker and runs flush/hydrate against it. rebuildMountEmbedAssets requires mountsMu held — the async goroutine acquires it itself (RegisterProjectMount holds mountsMu across attach, so spawning there is fine since the goroutine runs after the lock is released).
- The server-default-missing case (truly fresh box, zero models) still pulls synchronously on boot — by design, since we need that model's dim to size resident indexes and it's the documented one-time cost. If even that must be non-blocking later, you'd need deferred dim sizing (vecindex.New() takes no dim, so it's feasible) — out of scope here. The reported repro and acceptance were entirely the per-project-override case, which is now fully non-blocking.
