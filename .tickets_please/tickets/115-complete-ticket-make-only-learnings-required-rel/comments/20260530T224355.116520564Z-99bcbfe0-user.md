### Full learnings

- **Schema design: the load-bearing field test.** Of the three completion fields, only `learnings` is consumed across tickets (via `search_learnings`'s embeddings). `testing_evidence` + `work_summary` are within-ticket audit trail. Making the audit-trail pair optional removes the padding pressure (agents were writing "ran existing tests, all passed" / "one-line typo fix" to clear the ≥10-char gate, diluting the per-ticket embedding signal). General lesson: if a field doesn't earn its place across tickets, don't force it to be substantive on every ticket — make it optional.

- **The MCP envelope-recovery helper is now factored.** The `decodeArgEnvelope` private helper inside `internal/mcptools/tools.go` is the canonical entry point for any future handler that needs to read tool-call args while surviving the Claude Code `json.RawMessage` envelope bug. `requireStringArgs` and `optionalStringArgs` both build on it. If you add a tool that mixes required and optional string args, use both — don't reach back to `req.GetArguments()` directly or `RequireString`, both of which mis-handle non-map envelopes.

- **`mcp.Required()` is independent of svc-layer validation.** The tool registration's `Required()` only drives the JSON-schema check the client/server enforce on the wire. The svc layer has its own validation (`requireMinLen` etc.) which is what actually rejects empty/short values. They have to stay in sync — when you drop `Required()` you must also drop the svc check, or you get an inconsistent rejection (sometimes server-side, sometimes wire-side).

- **`completion.md` format is parsed back by `splitCompletionSections` (cache loader).** The exact heading strings `## Testing evidence`, `## Work summary`, `## Learnings` are matched literally on load. Dropping a heading when its content is empty is safe because the loader handles missing sections by leaving the corresponding `*string` as nil/empty — but if you ever add a fourth section, both the writer (this `CompleteTicket` function) and `splitCompletionSections` need the same heading string.

- **Avoid leaking optional fields into the `system_completion` comment body.** I rewrote both `completion.md` *and* the `system_completion` comment to omit empty sections — they share the same shape on disk, so a learnings-only completion produces a clean comment too. Without that, the comment would have rendered `Testing evidence:\n\nWork summary:\n\nLearnings:\n…` with blank-bodied labels, which looks broken in the web UI and the `ListComments` output.

- **Five doc surfaces had to be synced.** Tool registration description, `instructions.go` (the blob `initialize` returns every turn), `SPEC.md` (three separate mentions), project `CLAUDE.md`, `README.md`, and the user's global `~/.claude/CLAUDE.md`. The MCP `ServerInstructions` text is load-bearing — `TestServerInstructionsContainsLoadBearingPhrases` pins specific phrases (e.g. `complete_ticket`, `learnings`), so a rewrite must still contain them. If you ever shorten that blob, run the instructions_test first.

- **Build is currently broken on `main` by ticket 068's WIP.** `internal/web/components/layout/` is an entirely untracked directory containing a `flash.templ` whose generated `func Flash(f *Flash)` collides with the `Flash` struct in `data.go` (same package). This blocks `make check` and `make build`, but `go test ./internal/mcptools/... ./internal/svc/...` runs cleanly. Don't waste cycles diagnosing the layout failure on an unrelated ticket — it's ticket 068's to fix. The minimal fix when it lands is probably to rename the templ function (e.g. `FlashBanner`) so it doesn't shadow the struct.

- **The Claude Code MCP envelope bug bit this very completion.** First two attempts at `complete_ticket` with the full rich-content payload were rejected with `required argument(s) not found: work_summary, learnings` — same misleading error the global CLAUDE.md documents. Third attempt with short ASCII bodies succeeded. Even then, the saved `learnings` field on the ticket itself was tail-corrupted with `</learnings></invoke>` markers — harness wrapper bleeding into the field value. This is upstream in Claude Code; tickets_please server-side validation is fine (verified earlier by direct curl). When you hit it: retry once, then fall back to a minimal `complete_ticket` plus a follow-up `add_comment` for the rich content — exactly what this comment is.
