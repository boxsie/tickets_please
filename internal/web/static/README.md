# Web UI static assets

Files in this directory are baked into the binary via `go:embed` from
`../dev.go` and served under `/static/` by the web router. The `_src/` folder
holds the Tailwind v4 source used to regenerate `app.css` — Go's embed
deliberately skips `_`-prefixed entries so those sources stay out of the
binary.

## Files

| File              | Purpose                                                                                            |
|-------------------|----------------------------------------------------------------------------------------------------|
| `app.css`         | Compiled, minified stylesheet — embedded in the binary. Regenerated by `make css` / `make build`. |
| `htmx.min.js`     | Vendored htmx 1.9.12 (still in use during the templ migration).                                    |
| `htmx.LICENSE`    | 0BSD licence text for htmx.                                                                        |
| `_src/app.css`    | Tailwind v4 source: `@import "tailwindcss"`, `@theme` design tokens, `@layer components`.          |

## Regenerating `app.css`

`make build` runs `make generate` first, which runs `make css`, so a normal
build keeps the embedded CSS in sync with the source. To rebuild standalone:

```sh
make css            # one-shot
```

The build uses the [Tailwind v4 standalone CLI](https://tailwindcss.com/blog/standalone-cli)
— a single binary, no Node, no npm. If `tailwindcss` isn't on your PATH yet,
install once:

```sh
curl -sSL -o ~/go/bin/tailwindcss \
    https://github.com/tailwindlabs/tailwindcss/releases/latest/download/tailwindcss-linux-x64
chmod +x ~/go/bin/tailwindcss
```

(Pick the matching `tailwindcss-{darwin,linux,windows}-{arm64,x64}` asset.)

## Source layout (`_src/app.css`)

- `@import "tailwindcss"` — pulls in v4's preflight + utility generator.
- `@source "../../templates"` / `@source "../../components"` — tells v4 where
  to scan for class names so utility classes referenced in markup survive
  purge. `@layer components` blocks are emitted regardless of reference.
- `@theme { … }` — design tokens. Colours land as `--color-*` custom
  properties on `:root` AND power utility families like `bg-fg`, `text-accent`,
  `bg-badge-todo`. The four `--color-badge-*` tokens are the canonical column
  palette.
- `@layer base { … }` — element defaults (body font, link colour, headings).
- `@layer components { … }` — every semantic class the templates reference by
  name (`.topbar`, `.sidebar-link`, `.badge-todo`, …). This is what keeps
  the legacy `html/template` pages rendering correctly while we migrate to
  typed templ components on top of utility classes.

## Why a build step now?

Earlier in the project the README here said *"there is intentionally no
Tailwind step in the build pipeline"* and the regen was a hand-typed
`npx tailwindcss@3.4.10 …`. Phase 011 (Frontend modernisation foundation)
flipped that policy: we want Tailwind utilities composable with new templ
components, the standalone v4 CLI is a single binary so the toolchain ethos
holds, and `git diff --exit-code internal/web/static/app.css` after
`make build` is a clean drift check between source and output.

## Inline `style=` attributes

The migration ticket required removing hand-rolled CSS from inline `style`
attributes in templates. The only remaining inline styles are explicitly-
annotated *escape hatches* where the value is a server-computed percentage
(status-bar segment widths in `pages/projects/detail.tmpl`, phase progress
bar widths in `pages/phases/index.tmpl`). Anything else has either become a
utility class (`m-0`, `mt-4`) or a semantic class in `@layer components`
(`.logs-pre`, `.logs-meta`).
